Nonogram (Picross)
1. Тема проєкту
Застосунок дозволяє користувачу проходити логічну головоломку Nonogram (Picross) з вибором рівня складності, збереженням прогресу, таймером та перевіркою правильності розв’язку.

2. Постановка задачі
Цільовими користувачами застосунку є люди, які люблять логічні ігри та головоломки, а також хочуть тренувати уважність, пам’ять і логічне мислення. Основна проблема полягає в тому, що багато подібних ігор не мають зручної системи збереження прогресу, можливості продовжити незавершену гру або зрозумілого поділу за складністю. Через це користувачеві незручно повертатися до гри після перерви та відстежувати власний прогрес.

Метою застосунку є створення desktop-гри Nonogram (Picross), у якій користувач може обирати рівень складності(easy, medium, hard), проходити головоломки, перевіряти правильність виконання та зберігати результати. Користувач матиме можливість продовжити незавершений рівень або почати його заново. Очікуваним результатом є зручний та зрозумілий застосунок для проходження логічних головоломок із системою прогресу та збереження стану гри. Застосунок працюватиме офлайн, дані головоломок і прогрес користувача зберігатимуться локально у JSON-файлах, а платформою реалізації буде desktop-застосунок.

3. Сутності та дані
                У застосунку використовуються кілька основних сутностей, які відповідають за збереження інформації про користувача, головоломки та процес гри.

                3.1. Користувач (User)

                Ця сутність зберігає інформацію про гравця та його прогрес.

                Обов’язкові поля:

                id — унікальний ідентифікатор користувача;
                username — ім’я користувача;
                password — пароль.

                Опційні поля:

                best_time — найкращий час проходження;
                completed_puzzles — список пройдених рівнів.

                Спосіб ідентифікації: UUID.

                3.2. Головоломка (Puzzle)

                Містить інформацію про конкретний рівень гри.

                Обов’язкові поля:

                id — ідентифікатор головоломки;
                difficulty — рівень складності;
                grid_size — розмір сітки;
                solution_matrix — правильне рішення у вигляді матриці.

                Опційні поля:

                name — назва рівня;
                description — короткий опис.

                Спосіб ідентифікації: числовий ID.

                3.3. Ігрова сесія (GameSession)

                Відповідає за поточний процес проходження гри.

                Обов’язкові поля:

                id — ідентифікатор сесії;
                user_id — користувач, який грає;
                puzzle_id — вибрана головоломка;
                start_time — час початку гри;
                current_grid_state — поточний стан сітки.

                Опційні поля:

                end_time — час завершення;
                mistakes_count — кількість помилок.

                Спосіб ідентифікації: UUID.

                3.4. Результат (Result)

                    Зберігає інформацію про завершення гри та статистику проходження.

                    Обов’язкові поля:

                    id — ідентифікатор результату;
                    session_id — посилання на ігрову сесію;
                    time_spent — витрачений час;
                    is_completed — чи завершено рівень.

                    Опційні поля:

                    accuracy_percent — точність проходження;
                    checks_count — кількість перевірок правильності.

                    Спосіб ідентифікації: UUID.

4. Екрани застосунку

        1. Екран входу / реєстрації

        Призначення: вхід або створення акаунта.
        UI: поля логін і пароль, кнопки “Увійти” та “Реєстрація”.
        Дані та дії: введення даних користувача, авторизація.

        2. Екран вибору складності

        Призначення: вибір рівня гри.
        UI: кнопки Easy, Medium, Hard, список рівнів.
        Дані та дії: перегляд прогресу, вибір складності та рівня.

        3. Екран гри

        Призначення: проходження головоломки.
        UI: ігрова сітка, таймер, кнопки “Перевірити” і “Фініш”.
        Дані та дії: заповнення клітинок, перевірка правильності, завершення гри.

        4. Екран результату

        Призначення: перегляд результатів проходження.
        UI: час, кількість помилок, кнопка повернення.
        Дані та дії: перегляд статистики та повернення до списку рівнів.

5. Сценарії користувача
            1. Основний сценарій

            Відкрив застосунок → увійшов у акаунт → обрав складність → обрав головоломку → заповнив сітку → натиснув “Перевірити” → отримав результат.

            2. Створення ігрової сесії

            Обрав рівень → система створила нову GameSession → запустився таймер → користувач почав гру.

            3. Редагування / продовження гри

            Відкрив незавершений рівень → система завантажила збережений стан → користувач продовжив гру або почав заново.

            4. Завершення гри

            Натиснув “Фініш” → таймер зупинився → результат зберігся → рівень позначився як завершений або незавершений.

            5. Після MVP

            Пошук рівня → фільтрація за складністю → імпорт нових puzzle → експорт результатів.

6. Архітектура застосунку

            Архітектура застосунку
            UI (інтерфейс)

                    Відповідальність: показує всі екрани, приймає дії користувача (кліки, введення), передає команди в логіку, але не містить правил гри.

                    Приклади компонентів:
                    LoginScreen — вхід і реєстрація користувача
                    DifficultyScreen — вибір рівня (easy/medium/hard) і перегляд прогресу
                    GameScreen — ігрове поле Nonogram, кнопки “Перевірити”, “Фініш”, таймер
                    ResultScreen — показ результату (час, помилки, статус)
            Логіка (business layer)

                    Відповідальність: обробляє правила гри, перевіряє правильність заповнення сітки, керує станом гри (початок, пауза, завершення), рахує час і результат.

                    Приклади компонентів:
                    GameEngine — перевіряє, чи правильно заповнена сітка Nonogram
                    GameStateManager — зберігає стан гри (in progress / paused / finished / incomplete)
                    TimerService — запускає, зупиняє і рахує час гри
                    ProgressManager — зберігає, які рівні пройдені
            Дані (data layer)

                    Відповідальність: зберігає і завантажує всі дані застосунку, працює з JSON або локальною базою даних, передає дані в логіку.

                    Приклади компонентів:
                    UserRepository — збереження даних користувачів
                    PuzzleRepository — завантаження головоломок із JSON
                    SessionRepository — збереження ігрових сесій (прогрес, стан гри)
                    ResultRepository — збереження результатів проходження

7. MVP (мінімально життєздатний продукт)

            1. Ключовий end-to-end сценарій

            Користувач обирає головоломку → відкриває гру → заповнює сітку Nonogram → натискає “Перевірити” або “Фініш” → отримує результат.

            2. Мінімальний набір функцій (MVP)

            Без цих функцій застосунок не має сенсу:

            вибір рівня складності
            завантаження головоломок з JSON
            ігрове поле Nonogram
            перевірка правильності рішення
            таймер
            збереження результату гри
            3. Функції після MVP

            реєстрація та профіль користувача
            система пройдених рівнів (прогрес)
            продовження незавершеної гри
            рейтинг або таблиця лідерів
            підказки під час гри
            темна/світла тема
            система досягнень
            експорт/імпорт головоломок

8. Критерії готовності MVP

        Обов’язкові екрани

        Для готового MVP мають бути реалізовані такі екрани:

            екран вибору складності та головоломки
            екран гри (Nonogram поле)
            екран результату
        Сценарії, які повинні працювати без помилок
            користувач може обрати головоломку і запустити гру
            користувач заповнює сітку і може перевірити правильність рішення
            користувач завершує гру через “Перевірити” або “Фініш”
            після завершення показується результат (час, помилки, статус)
            гра не зависає і не втрачає введені дані під час проходження
        Дані, які повинні зберігатися і відновлюватися

        Після перезапуску застосунку мають зберігатися:

            прогрес поточної гри (стан сітки Nonogram)
            інформація про обрану головоломку
            час початку або загальний час проходження
            результат завершених ігор (час, статус “пройдено/не пройдено”)
